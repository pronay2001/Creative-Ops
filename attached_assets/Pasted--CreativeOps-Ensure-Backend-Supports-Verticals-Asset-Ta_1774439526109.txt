## CreativeOps: Ensure Backend Supports Verticals, Asset Taxonomy & Timesheet CTAs

The frontend (app.js and seed-data.js) already has these features implemented. This prompt ensures the backend schema, API, and data layer properly support them.

---

### 1. Vertical Field Support

The `requests` table needs a `vertical` column. Ensure the schema includes:

```sql
ALTER TABLE requests ADD COLUMN IF NOT EXISTS vertical TEXT;
```

The `POST /api/requests` and `PATCH /api/requests/:id` endpoints must accept and store the `vertical` field.

The `GET /api/load-all` response must include the `vertical` field on each request object.

The verticals list is static (defined in seed-data.js, not stored in DB):
- hoichoi, hoichoi Studios, hoichoi TV+, hoichoi FM, Sooper, Sooper FM, LoglineAI, SVF

### 2. Asset Type Taxonomy (Three Categories)

### 3. Timesheet CTA (Clock-In/Clock-Out) Backend Support

The frontend has Start / Shift Job / Start Again / End buttons for time tracking. The backend needs to support this clock-based workflow.

Add a `timesheet_clock` table:

```sql
CREATE TABLE IF NOT EXISTS timesheet_clock (
  id SERIAL PRIMARY KEY,
  user_id TEXT REFERENCES users(id) NOT NULL,
  request_id TEXT REFERENCES requests(id),
  started_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  ended_at TIMESTAMPTZ,
  duration_minutes NUMERIC(6,2),
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

Add these API endpoints:

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/timesheet/clock-start` | Start tracking. Body: `{ requestId }`. Creates a new clock entry with `started_at = NOW()`. If there's already a running clock for this user, auto-end it first. |
| POST | `/api/timesheet/clock-shift` | Shift to new task. Body: `{ requestId }`. Ends current clock entry and starts a new one for the new request. |
| POST | `/api/timesheet/clock-resume` | Resume after break. Body: `{ requestId }`. Same as clock-start but explicitly for resuming. |
| POST | `/api/timesheet/clock-end` | End tracking. Ends the current running clock entry, calculates `duration_minutes` from `started_at` to now. |
| GET | `/api/timesheet/clock-status` | Get current clock state for the logged-in user. Returns: `{ isRunning, currentEntry, requestId, startedAt, elapsedMinutes }` |
| POST | `/api/timesheet/clock-reassign` | Manager only. Body: `{ clockEntryId, newUserId }`. Reassign a clock entry to a different user. Only allowed for `creative_lead` role. |

The clock-end action should also auto-create a corresponding entry in the `timesheet_entries` table (the existing manual hour-logging table) so everything stays in sync.

### 4. Seed Data Alignment

When running `db/seed.js`, ensure:
- All request records use the NEW asset type IDs (not old ones)
- The `vertical` field is populated on seed requests (default to `'hoichoi'` if not specified)
- Deliverable records also use new asset type IDs

If any seed request references an asset type ID that doesn't exist in `seed-data.js`, map it to the closest new equivalent using the table above.

Build these changes into the existing backend. Don't modify the frontend files (app.js, style.css, seed-data.js) as they already have these features implemented.
